## What this app is

`doubleclicker-1` is the **multi-tenant blog frontend + provisioner**. One
codebase, many Fly.io apps. Each tenant (e.g. `thevillageway`, `valentia`,
`modernlongevity`) gets a clone of the same Next.js app deployed as its own
Fly app, configured by tenant-specific Fly secrets and a shared Supabase DB.

Position in the broader ecosystem:

```
                  ┌─────────────────────────┐
                  │  doubleclicker-1        │  ← THIS REPO
                  │  (provisioner +         │     - admin UI
                  │   tenant blog frontend) │     - /api/provision
                  └────────────┬────────────┘     - tenant blog rendering
                               │
              POST auto-onboard│ x-provision-secret
                               ▼
                  ┌─────────────────────────┐
                  │  doubleclicker.fly.dev  │  ← orchestrator
                  │  (content pipeline)     │     - keyword discovery
                  └────────────┬────────────┘     - topical map (v3 hat research)
                               │                  - outline + write + publish
              shared Supabase  │
                               ▼
                  ┌─────────────────────────┐
                  │  stitch-app.fly.dev     │  ← writer engine
                  │  (Fichtean + DeBono)    │     - audience research
                  └─────────────────────────┘     - per-format adapters
```

  to `app_settings.static_pages.{username}`

## How a new brand gets provisioned (today)

This is the current end-to-end. The "auto-call ingester" change is **in
progress on the user's side** — once shipped, steps 1+2+3 collapse into a
single API call.

### Step 1 — Optionally ingest a brand bundle (markdown repo)

```bash
SECRET=...
GH_TOKEN=...   # only needed for private repos (or set GITHUB_TOKEN env var)

curl -X POST https://doubledoubleclickclick.fly.dev/api/admin/ingest-brand-bundle \
  -H "Authorization: Bearer $SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "newbrand",
    "bundle_repo_url": "https://github.com/owner/newbrand-content",
    "github_token": "'$GH_TOKEN'",
    "dry_run": false
  }'
```

Returns the extracted `static_pages` payload (founder_story, philosophy,
immutable_rules, mission_long, menu_items, custom_pages) and writes it to
`app_settings.static_pages.{username}`.

### Step 2 — Provision the site

```bash
curl -X POST https://doubledoubleclickclick.fly.dev/api/provision \
  -H "Authorization: Bearer $SECRET" \
  -H "Content-Type: application/json" \
  -d @provision-payload.json
```

11 phases run automatically:
1. **db_seed** (critical): brand_guidelines + brand_specifications +
   company_information + authors + integration_credentials + target_market
   (if ica_profile) + brand_image_styles (if image_style) + 4 app_settings
   rows (publishing_settings, onboard_config, network_partners, v3 flags)
   + static_pages (if provided)
2. **auto_onboard** (important): fires `POST {DC_URL}/api/strategy/auto-onboard`
   to trigger the content pipeline (skipped if `skip_pipeline=true`)
3. **2.5 hero_image + favicon** (silent): nano-banana-pro hero (with GPT-4.1
   prompt enhancement) + Ideogram-transparent logo derivative as favicon
4. **google_services** (important): GA4 property + GTM container
5. **fly_deploy** (critical): creates `{username}-blog` Fly app from base
   image, sets all brand identity as Fly **secrets** (so they survive
   redeploys), allocates IPv4+IPv6, spawns machine
6. **domain_purchase** (optional): registers domain via Cloud Domains
7. **tls_certs** (optional): requests Fly ACME certs for `{domain}` and
   `www.{domain}`
8. **search_console** (optional): adds `sc-domain:{domain}` + returns DNS
   TXT verification token
9. **dns_config** (optional): auto-configures Cloud DNS records
10. **email_notify** (optional): emails `contact_email` from
    `noreply@sewo.io` (Resend) with full DNS records table
11. **analytics_log** (silent): inserts `site_provisioned` event

### Step 3 — Trigger pipeline manually if `skip_pipeline=true`

```bash
curl -X POST https://doubleclicker.fly.dev/api/strategy/auto-onboard \
  -H "x-provision-secret: $SECRET" \
  -H "Content-Type: application/json" \
  -d '{"username":"newbrand"}'
```

### Step 4 — Once topical map completes, schedule articles

`api/strategy/launch.js` step 9 (`stepCreateSchedule`) does this
automatically as part of the chain. Manual fallback:

```bash
curl -X POST https://doubleclicker.fly.dev/api/strategy/create-schedule \
  -H "x-provision-secret: $SECRET" \
  -H "Content-Type: application/json" \
  -d '{"username":"newbrand","articles_per_day":1}'
```

Articles are spread across N days (one per day at `articles_per_day=1`)
via the `content_schedule` table. The doubleclicker `daily-writer` cron
picks them up.

## Architecture decisions worth preserving

1. **Brand identity lives as Fly secrets, not machine env**. Discovered the
   hard way when a `fly deploy` wiped TVW's `NEXT_PUBLIC_BRAND_USERNAME`
   and the site briefly served as the wrong tenant. Provisioner Phase 4
   sets BRAND_USERNAME, SITE_URL, SITE_NAME, CONTACT_EMAIL, GA_ID, GTM_ID,
   LANGUAGES (and NEXT_PUBLIC_ mirrors) via `fly.setSecrets()`. Survives
   any future `fly deploy`.

2. **Fly Depot builder is unreliable**. When `fly deploy` hangs on
   "Waiting for depot builder...", use `--depot=false` to fall back to
   the legacy remote builder. Documented but worth saying again — both
   our deploys tonight failed without `--depot=false`.

3. **DeBono × Schwartz primitives are duplicated** in stitch + doubleclicker
   (identical 347-line file). The ENGINE only lives in stitch
   (`audienceResearchEngine.js`, 813 lines). doubleclicker has a fork
   `topicResearchEngine.js` (477 lines) for per-theme research during the
   v3 topical map. doubleclicker-1 (this app) does NOT need to implement
   either — when the brand-level research call gets wired in, it should
   call stitch's `/api/brands/research` and read the result from the
   shared DB.

4. **Tier 1 + Tier 2 + Tier 3 static page content** (designed but not all
   shipped):
   - **Tier 1 (shipped)**: Bundle markdown extracted by Opus →
     `app_settings.static_pages`. Founder voice verbatim where present.
   - **Tier 2 (designed, not shipped)**: When no bundle, generate
     static_pages from `brand_guidelines.voice_and_tone` via Claude
     Opus. ~30s, no DeBono dependency, no race condition with stitch.
   - **Tier 3 (designed, not shipped)**: When stitch's brand-level
     audience research completes, regenerate static_pages with hat ×
     awareness framing. Async webhook from stitch. Replaces tier 1/2
     content in-place.

5. **v3 is opt-in at the doubleclicker level via three `app_settings`
   flags**. Provisioner always seeds them (no opt-out). Existing v2
   tenants are unaffected.

6. **Claude Code injects `ANTHROPIC_API_KEY=""` (empty)** into shell
   environments, which silently overrides `.env.local`. Local dev
   workaround:
   ```
   ANTHROPIC_API_KEY=$(grep '^ANTHROPIC_API_KEY=' .env.local | cut -d= -f2) npm run dev
   ```
   Not a problem in production (Fly secrets ARE the process env).

## Quick reference: useful commands

```bash
# Local dev (note the ANTHROPIC_API_KEY override for Claude Code envs)
ANTHROPIC_API_KEY=$(grep '^ANTHROPIC_API_KEY=' .env.local | cut -d= -f2) npm run dev

# Deploy base (provisioner)
export FLY_API_TOKEN=$(grep '^FLY_API_TOKEN=' .env.local | cut -d= -f2-)
flyctl deploy --remote-only                # try depot first
flyctl deploy --remote-only --depot=false  # fallback if depot is down

# Deploy a specific tenant
flyctl deploy -a thevillageway-blog --remote-only

# Provision a new brand (after seeding bundle if you have one)
SECRET=$(grep '^PROVISION_SECRET=' .env.local | cut -d= -f2)
curl -X POST https://doubledoubleclickclick.fly.dev/api/provision \
  -H "Authorization: Bearer $SECRET" \
  -H "Content-Type: application/json" \
  -d @payload.json

# Trigger doubleclicker auto-onboard manually
curl -X POST https://doubleclicker.fly.dev/api/strategy/auto-onboard \
  -H "x-provision-secret: $SECRET" \
  -H "Content-Type: application/json" \
  -d '{"username":"newbrand"}'

# Pipeline status for a brand
```
